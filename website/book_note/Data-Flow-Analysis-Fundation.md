# Data Flow Analysis Fundation

数据流分析并不是直接对源代码做推理，而是通常建立在一种更规整的中间表示之上。  


## IR 概念以及为什么使用 IR

静态分析的目标，是在 **不真正运行程序** 的前提下，推断程序在不同程序点上的性质，例如：

- 一个变量当前可能保存哪些值
- 一个对象可能由哪些变量引用
- 某个定义是否会在后续被使用
- 某段数据是否可能传播到危险位置

如果直接在源代码上分析，会遇到一个问题：源代码保留了太多“写法层面”的信息，不同语言构造的表面差异很大，但它们背后的执行语义往往相似。  
因此编译器通常会先把源程序翻译成更适合分析和优化的 **Intermediate Representation, IR**，静态分析也往往建立在 IR 之上。

可以把一个典型的编译流程理解为：

- `Scanner`：把 source code 切分成 token
- `Parser`：根据语法规则把 token 组织成 AST
- `Semantic Analyzer / Type Checker`：补充类型和语义信息
- `Translator`：把 AST 翻译成 IR
- `Code Generator`：再把 IR 翻译成机器码

其中对静态分析最关键的一步，就是把高级语言结构翻译成一种更统一、更显式的表示形式。  
这一步做得越规整，后续分析就越容易自动化。

## 三地址码（Three-Address Code）

课程中最常见的一类 IR 就是 **Three-Address Code, 3AC**。

所谓“三地址码”，是说一条指令通常至多包含三个地址：

- 一个目标地址
- 一个或两个源地址

这里的“地址”不一定真的是内存地址，它更常表示普通变量， 常量， 编译器生成的临时变量

常见的三地址码形式有：

```text
x = y
x = y op z
x = op y
if x goto L1
goto L2
return x
```
如果不是三地址码也可以拆分为三地址码

![Three-address code example](./img/three-address-code-example.avif)

这种拆分非常重要，因为它把一个复杂表达式变成了若干个简单步骤。  
后续无论做常量传播、活跃变量分析，还是可达定义分析，都会更容易逐条处理。

## 3AC 为什么适合数据流分析

从数据流分析的角度看，3AC 有几个明显优点：

- 每条语句的定义和使用关系更清晰
- 复杂表达式被拆开后，传递函数更容易设计
- 控制跳转被显式表示，便于构造 CFG
- 临时变量让中间结果变得可追踪

比如在活跃变量分析中，我们最关心的是每条语句的：

- `def`：它定义了谁
- `use`：它使用了谁

如果程序已经被拆成三地址码，那么 `def/use` 往往能直接从语句表面读出来，这就是后续数据流方程能够成立的基础。

## Static Single Assignment (SSA)

另一种非常重要的 IR 形式是 **Static Single Assignment, SSA**。

SSA 的核心要求是：**每个变量只能被赋值一次**

这对很多静态分析和优化都很有帮助。

## Basic Block

在数据流分析里，我们通常不会把程序看成“单纯的一串语句”，而会进一步组织成 **Basic Block**。

基本块可以理解为满足下面性质的一段连续代码：

- 只能从第一条语句进入
- 只能从最后一条语句离开

也就是说，基本块内部没有中途跳入，也没有中途跳出。  
一旦进入这个块，语句就会顺序执行到结尾。

![Basic block example](./img/basic-block-example.avif)

这样划分的好处是：

- 可以把若干线性执行的语句打包成一个节点
- 简化程序结构表示
- 方便在块级别或语句级别定义数据流方程

通常一条基本块会以以下语句作为边界：

- 条件跳转
- 无条件跳转
- 返回语句
- 可能成为跳转目标的语句

## Control Flow Graph

把基本块连起来，就得到后续所有数据流分析几乎都会依赖的 **Control Flow Graph, CFG**。

CFG 是一个图：

- 节点通常是基本块，或者更细粒度的 IR 语句
- 边表示控制可能从一个节点流向另一个节点

一个典型 CFG 还会包含：

- `Entry`：统一入口
- `Exit`：统一出口

![CFG example](./img/cfg-example.avif)

例如，一个简单条件语句：

```text
if (x > 0) {
    y = 1;
} else {
    y = 2;
}
z = y + 1;
```

在 CFG 中就会出现：

- 条件判断处分叉
- 两条分支分别执行
- 在 `z = y + 1` 之前重新汇合

这正是数据流分析最关心的场景，因为程序性质往往需要在“不同路径的信息”之间进行传播和合并。
